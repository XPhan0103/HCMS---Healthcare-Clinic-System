1.1 Software Architecture

Kiến trúc Phần mềm Tổng thể — Modular Monolith

The HCMS system adopts a Modular Monolith architecture pattern as defined in ADR-001. This pattern provides clear bounded-context separation at the package level while maintaining a single deployable unit — optimized for the project's lean operational scale (1-2 doctors, 10-20 patients/day, 50-500 CCU) and a small development team (1 Tech Lead + 2-4 developers).

1.1.1 Architecture Principles

Separation of Concerns Each layer has a single, well-defined responsibility. UI logic, business logic, data access, and cross-cutting concerns are strictly isolated.
Module Encapsulation Each business module (Patient, Appointment, etc.) owns its controller → service → repository → entity stack. Inter-module calls only through service interfaces.
Shared Kernel Common DTOs, exceptions, utility classes, and base entities are centralized in the common package to avoid duplication while maintaining consistency.
Stateless Communication REST API with JWT-based stateless authentication. No server-side session state. Horizontal scalability without sticky sessions.

1.1.3 Layer Responsibilities

Layer Technology Responsibility
Presentation React 19, Vite 8, Tailwind CSS, Shadcn/ui, TypeScript Renders the user interface (SPA). Handles client-side routing, form validation, state management (Zustand), and API communication via Axios with JWT token interceptors. Separate deployment on Vercel/Netlify.
API Gateway Spring Boot 3.x, Spring Security, JWT, OAuth 2.0 Single entry point for all REST API requests. Handles authentication (JWT filter), authorization (role-based access control), CORS policy enforcement, request validation, and auto-generated API documentation (Swagger UI).
Module Java 21, Jakarta Validation, Spring Services Contains all domain-specific business logic organized into bounded contexts. Each module (auth, patient, appointment, consultation, billing) owns its full vertical slice: Controller → Service → Repository → Entity → DTO. Inter-module communication occurs only through service interfaces.
Data Spring Data JPA, Hibernate, PostgreSQL 17, Flyway Manages data persistence, ORM mapping, and schema versioning. PostgreSQL provides ACID transactions, JSON support, and encryption at rest for EMR data. Flyway ensures reproducible database migrations.

1.1.4 Module-to-Business Goal Mapping

Module Core Entity Business Goal (Ref) Key Functions
auth User / Role Cross-cutting (Security) User registration/login, JWT issuance, role-based access (DOCTOR, RECEPTIONIST, PARENT), OAuth 2.0 Google integration
patient Patient G2: Digitize Pediatric EMR Patient demographics, parent contact info, basic medical history, encrypted EMR records
appointment Appointment G1: Self-Service Booking Portal Online booking portal, time slot management, doctor assignment, status tracking (Pending → Confirmed → Completed → Cancelled)
consultation Visit, Prescription G2: Digitize Pediatric EMR Clinical visit records, symptoms/diagnosis capture, prescription generation, clinical notes
billing Billing G3: Streamlined Checkout Invoice auto-generation from consultation, fee calculation, 1-click payment status update (Paid/Unpaid), no insurance logic

Architecture Decision Reference: This design strictly adheres to ADR-001: Adopt Modular Monolith with Spring Boot + React Tech Stack. The modular monolith was selected over Microservices (rejected — over-engineered for 50-500 CCU) and Serverless (rejected — cold-start latency unacceptable for booking UX). Module boundaries are enforced via Java packages with package-private access modifiers to prevent cross-module violations.

1.2.2 Package Dependency Rules

# Rule Rationale
1 All modules may depend on common (Shared Kernel) Shared DTOs, base entities, exception handlers and utilities are centralized to avoid code duplication across modules.
2 No module may directly access another module's repository or entity Prevents tight coupling. All inter-module data access must go through the target module's service interface.
3 Use package-private access on repositories and entities by default Java compiler enforces module boundaries. Only public service interfaces are exposed to other modules.
4 The config package is cross-cutting and accessible by all modules Security filters, CORS rules, and Swagger configuration apply globally to all API endpoints.
5 Frontend features/ mirror backend modules for naming consistency Reduces cognitive overhead. Developers can locate frontend code for any backend module instantly.

1.2.3 Inter-Module Dependency Matrix

Source Module Target Module Dependency Type Description
appointment patient Service call Booking requires patient lookup for appointment association.
appointment auth Service call Needs doctor (User with DOCTOR role) for assignment to appointments.
consultation patient Service call Visit records reference patient EMR data for medical history context.
consultation appointment Service call A consultation is triggered from a confirmed appointment. Status update on completion.
billing consultation Service call Invoice is auto-generated from consultation records (fee calculation from visit + prescription).

Design Constraint: All inter-module dependencies are unidirectional. Circular dependencies between modules are strictly prohibited. If a bidirectional data flow is required (e.g., appointment status update after consultation), use event-driven callbacks or DTOs passed through the Shared Kernel — never direct cross-module repository access.
